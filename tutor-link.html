TutorLink

A peer-tutor marketplace built end-to-end in one semester by a five-person team at San Francisco State. Students post requests, tutors post availability, courses tie everything together.

Next.js Express Knex SQL BFF pattern Team of 5

The idea

SFSU's CSC648 software-engineering course is a semester-long project run with a real team — five students, no professor in the trenches with you. We picked a tutoring marketplace because the domain was familiar (every CS undergrad has either offered or needed help) and the data model was rich enough to exercise everything we needed to learn.

What I built

  • Backend-for-Frontend architecture — Next.js front end with API routes that proxy a separate Express backend. The client never talks directly to the database, which made auth and validation easier to centralize.
  • Database layer — Knex for migrations and seed data. Schema covers courses, tutor posts, requests, and tutor applications. Reusable seeds let any teammate spin up a working dataset locally in seconds.
  • Route handlers — Express routers for each resource (courses, tutorPosts, requests, tutorApplications). Health endpoint for the BFF proxy layer.
  • Team coordination — translation from the Navy "ops LPO" instinct: clear ownership, lightweight standups, and a milestone-based deployment pipeline. Five people shipped on time, without an all-nighter the night before.

The "wow"

The BFF pattern isn't novel, but seeing it work on a five-person student team — keeping the front-end devs unblocked when the backend was in flux and vice versa — was the thing that made the project actually ship instead of stalling at week eight.
