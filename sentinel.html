Sentinel Shield

Mobile threat defense that blocks malicious DNS and URLs in real time — then turns every blocked event into an interactive security lesson and machine-verifiable compliance evidence mapped to NIST SP 800-171.

React Native Expo TypeScript Swift Network Extension iOS App Groups expo-router React Query NIST SP 800-171 Hash-chain evidence

The idea

Most security tools tell you something was blocked and move on. Sentinel Shield treats every blocked threat as a teachable moment and a compliance data point. A phishing link blocked → a micro-lesson on why that domain pattern is dangerous → a signed, timestamped evidence record tagged to the relevant NIST control. The same mobile device that faces the threat also learns from it and proves it.

The target audience is security-conscious individuals and small organizations under compliance obligations — contractors, consultants, and anyone who needs to demonstrate security awareness and posture without a dedicated security team.

What I built

  • Shield screen — real-time protection status, today's block counts broken down by threat kind (phishing, malware, C2, tracker), posture score, and navigation to Activity and Evidence. A "Simulate incoming threat" button injects a realistic blocked event from a curated pool so the protect → learn → evidence loop is fully demonstrable without the native filter running.
  • Activity screen — every blocked event expands into a control-tagged lesson: the blocked host, why it was flagged, a practical security habit the user should adopt, and the NIST SP 800-171 control it maps to. Tapping "Got it" marks the lesson learned and logs an awareness evidence record.
  • Evidence screen — control-mapped posture view. Each evidence row is signed and hash-chained to the previous record, creating a tamper-evident log that can be shared with a compliance officer or auditor without exposing browsing history. The privacy-preserving note explains what is and isn't in the evidence chain.
  • Native filter boundary — the DNS/URL content filter is a native iOS Network Extension, not JavaScript. The seam is lib/networkExtension.ts, which calls a native Swift module (SentinelFilter) exposing start(), stop(), getStatus(), and an onThreatBlocked event. The JS app runs fully without the native layer via graceful no-ops — the entire UI is functional today using the simulator path.
  • On-device reputation set — the Swift filter checks each connection against a Bloom filter shipped in the app bundle, updated server-side. Blocked events are appended to a shared App Group store and consumed by the React Native layer via the native module event.
  • Hash-chain evidence — lib/evidence.ts signs and chains evidence records cross-platform using expo-crypto. Each record includes a SHA-256 digest of the previous record's hash, a timestamp, the control ID, and a summary — no host or URL in the chain. In production the signDigest() call swaps to a Secure Enclave key and App Attest without changing the chain format.

The Cognivate connection

Sentinel Shield is built to drop straight into the same monorepo as Cognivate. It reuses the same component conventions (Card, PrimaryButton), the same theming system (constants/colors.ts, hooks/useColors.ts, lib/design/tokens.ts), the same auth/billing seam in _layout.tsx, and the same API client pattern (customFetch, React Query hooks). Anything I shipped in Cognivate — haptics, Reanimated press states, expo-router file-based routing, EAS Build pipeline — carries over for free.

What makes it interesting

The native boundary is the hard part. iOS Network Extension runs in a separate process with a small memory budget and no access to the UI thread. Shipping blocked events to the React Native layer requires an App Group shared container, a polling or notification bridge, and careful memory discipline in Swift so the extension doesn't get killed mid-connection. The JS side is intentionally unaware of those details — it just consumes events from the native module interface.

The hash-chain design is deliberately minimal. It doesn't prove what the user did; it proves that a sequence of events happened in order and haven't been tampered with since they were recorded. That's enough to satisfy an auditor asking "show me your security awareness activity" without requiring users to hand over a browsing log.
